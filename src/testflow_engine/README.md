# testflow_engine 说明

`testflow_engine` 是新的测试执行引擎。

如果你第一次看这个目录，可以先记住一句话：

**这里解决的是“怎么执行测试”，不是“怎么管理测试”。**

也就是说：

* 平台负责管理用例、套件、环境、任务、运行记录
* 引擎负责把一个“标准化用例”真正执行出来

---

## 1. 新手第一次进入这里，先做什么

建议不要一上来就直接翻源码，先跑这两个命令：

```bash
cd /Users/ys/PyCharmProject/PytestAutoApi
source .venv/bin/activate
make test-engine
```

如果你还想顺手确认兼容层：

```bash
source .venv/bin/activate
python -m pytest tests/compat -q
```

原因很简单：

* `tests/engine/` 已经把当前引擎最核心的执行链路串出来了
* `tests/compat/` 能帮你快速理解“旧 YAML 如何进新模型”

对新手来说，从测试反推结构，通常比直接看实现更省时间。

---

## 2. 为什么要单独做一个引擎

旧框架里最难维护的一点，是很多职责混在一起：

* YAML 解析
* 动态变量替换
* 请求发送
* 依赖提取
* 断言
* 报告
* 平台逻辑

这些东西一旦混在一起，后面无论改平台还是改执行链路，都会互相牵连。

新的做法是把执行核心先抽出来，形成独立的“引擎层”。

这样平台、worker、调度器、未来的 CLI，都可以复用同一套执行逻辑。

---

## 3. 目录职责

### `models/`

这里定义的是执行过程中的标准数据模型。

比如：

* 一个 run 长什么样
* 一个 case 长什么样
* 一个请求长什么样
* 一条断言长什么样
* 一次执行结果怎么表达

对新手来说，可以把它理解成：

**这是新引擎的“通用语言”。**

只要外部输入最终能转换成这些模型，引擎就能执行。

### `executor/`

这里是执行主链路。

重点看：

* `engine.py`：执行 run 和 case 的主流程
* `rendering.py`：模板变量渲染
* `transport.py`：发送 HTTP 请求的抽象层
* `scheduler.py`：轻量调度/排队骨架

注意：

`engine.py` 不应该直接绑死在 `requests` 或某个具体平台接口上。
所以它会依赖 `TransportAdapter` 这种抽象，而不是自己直接写网络请求细节。

### `assertions/`

这里处理断言。

断言层的职责非常单纯：

* 从响应或上下文里取值
* 按操作符比较
* 产出统一的断言结果

以后如果要新增断言操作，优先在这里扩展，而不是把比较逻辑散落到各处。

### `plugins/`

这里是扩展点。

平台只覆盖 80% 常规接口场景，复杂逻辑不会强行做成低代码。
所以这里会承接：

* 特殊鉴权
* 签名算法
* 前后置逻辑
* 自定义断言

你可以把插件理解成：

**在不破坏主执行链路的前提下，给复杂业务开“合法后门”。**

### `reporting/`

这里负责聚合执行结果。

第一阶段它还只是轻量骨架，但后续会逐步接：

* 数据库存储
* 平台结果页
* Allure 归档
* 趋势分析

### `compat/`

这里专门负责兼容旧资产。

当前最关键的是 `legacy_yaml`，它的职责不是永久保留旧框架，而是：

* 读取老 YAML
* 转换成新模型
* 找出哪些字段能自动迁移
* 标记哪些逻辑后续要插件化或人工整理

---

## 4. 最重要的执行链路

当前引擎的主流程非常刻意地保持简单：

```text
run -> case -> transport -> assertion -> report
```

这样做的原因是：

1. 先把最稳定的主干固定下来
2. 再把变量渲染、插件、报告、兼容层这些能力逐步挂上去

如果一开始就把所有历史能力一次性搬进来，很容易又回到旧框架那种“大杂烩”状态。

---

## 5. 推荐阅读顺序

如果你想最快理解引擎，建议按这个顺序看：

1. `tests/engine/test_engine_skeleton.py`
   先看“最小闭环到底长什么样”
2. `models/core.py`
   再理解“输入和输出长什么样”
3. `executor/engine.py`
   看主执行流程怎么跑
4. `executor/rendering.py`
   看请求变量怎么渲染
5. `executor/transport.py`
   理解为什么请求发送要抽象
6. `assertions/engine.py`
   看断言如何统一计算
7. `plugins/base.py`
   理解复杂逻辑未来怎么接进来
8. `compat/legacy_yaml.py`
   理解旧 YAML 如何桥接到新引擎
9. `reporting/collector.py`
   看结果目前怎么收集

如果你现在的任务是迁移旧 YAML，建议把 `tests/compat/test_legacy_yaml_compat.py` 也一起看掉。

---

## 6. 从测试里能快速看懂什么

### `tests/engine/test_engine_skeleton.py`

这个测试最值得先看，因为它已经把最小闭环跑通了：

* 定义 run
* 定义 case
* 注入 transport
* 执行断言
* 收集 report

它能帮你快速建立一个非常重要的感觉：

**引擎当前最在乎的是“链路清晰”，不是“功能一次性做全”。**

### `tests/compat/test_legacy_yaml_compat.py`

这个测试能帮你理解另一个重要问题：

**旧资产不是直接丢掉，而是先桥接进新模型。**

这里能看到：

* 历史 YAML 如何读取
* 哪些字段能直接迁
* 哪些字段先暂存到 extras

---

## 7. 这阶段还没有完成什么

当前引擎还只是从“骨架”进入“第一条真实链路”的阶段，所以有些东西还没做完：

* 真实 HTTP transport 还在继续补强
* 变量渲染能力还只是第一阶段
* legacy YAML 兼容还没覆盖全部旧语法
* reporting 还没接数据库和平台详情页
* 调度还不是生产级队列系统

这不是缺陷，而是刻意分阶段。

目标是先把结构立稳，再逐步扩能力。

---

## 8. 新手最容易误解的地方

### 误解 1：兼容层存在，就说明旧 YAML 会永久保留

不是。

`compat/legacy_yaml.py` 的目标是：

* 帮我们迁移
* 帮我们识别缺口
* 帮我们减少一次性切换风险

而不是让两套模型永远并存。

### 误解 2：引擎目录里应该顺手放平台逻辑

不应该。

只要某段逻辑开始带上：

* 页面语义
* 平台数据库语义
* UI 交互语义

就应该优先警惕它是不是放错层了。
